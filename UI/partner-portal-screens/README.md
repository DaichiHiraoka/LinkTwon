# LinkTwon パートナーポータル 画面デザイン

イベント受付（主催者向け）と商品交換（商店向け）の全画面 HTML モックアップ。
**形状・余白・タイポはエンドユーザー UI と同じ**、**色は赤基調**でエンドユーザー UI（緑基調）と視覚的に区別。

## 設計方針

1. **商店もイベント主催者もユーザー** — 業務システムではなく、エンドユーザーアプリと同じ温度感
2. **色のみ赤基調にして "別のアプリ" であることを一目で識別** — エンドユーザー（緑） vs パートナー（赤）
3. **操作説明文は最小限** — UI 自体で何をすればよいか分かる。テキストでの解説はしない
4. **各画面は 1 つの主要アクションに集中** — 迷わせない
5. **大型タッチターゲット** — ボタン 88px、入力 80px、フォント 22px 以上

## アプリ識別

- **エンドユーザー UI = 緑基調** (`#2E7D32`)
- **パートナーポータル = 赤基調** (`#DC2626` / `#E11D48`)
- パートナー同士の識別:
  - **イベント受付**: 鮮やかな赤 `#DC2626`
  - **商品交換**: ローズ赤 `#E11D48`（`<body class="theme-store">`）
  - 同じ「赤ブランド」内のサブシェード違い + 見出しテキスト + サムネ絵文字で識別

## エラー UI の方針

過去の設計では 6 種のエラーを 1 画面に並べる「ドキュメント的」な作りだったが、これは UI ではなくドキュメント。
新設計では:

- 一つのエラー画面パターン（アイコン + 短文 + アクション）を `08-error-states.html` で代表表示
- 他のエラー種（重複、形式不正、対象外、ポイント不足、通信障害...）はすべて同じパターンで表現
- 実装時にメッセージとアイコンだけ差し替える

代表的なエラーケースとメッセージ例:

| ケース | アイコン | メッセージ |
|---|---|---|
| QR期限切れ | ⏰ | QRの有効期限が切れています |
| ポイント不足 | 💰 | ポイントが足りません |
| 重複受付/交換 | ♻ | この方は既に受付済みです / 既に交換済みです |
| 形式不正 | ⚠ | QRが読み取れませんでした |
| 通信エラー | 📡 | 接続できませんでした |

## 実装側との対応

`partner-portals/server.js` のレスポンスを各画面が受ける想定:

| API | 画面 |
|---|---|
| `GET /api/bootstrap` | E-01〜E-04 / S-01〜S-04 |
| `POST /api/event/check-ins` | E-06 → E-07 (成功) / E-08 (失敗) |
| `POST /api/store/exchanges` | S-06 → S-07 (成功) / S-08 (失敗) |

QR ペイロード形式は `linktown://user-present?...` を維持。

## 閲覧

```powershell
start UI\partner-portal-screens\index.html
```

[index.html](index.html) から全 16 画面に飛べる。各画面は `.tablet-frame` 内に 2100×1300 で描画され、CSS で 60% 縮小して表示。
