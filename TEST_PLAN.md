# TEST_PLAN

`FEATURE_IMPLEMENTATION_PLAN.md` を元にしたテスト計画です。  
テストは、ユーザーまたは管理者が GUI で確認するものと、Codex が自動で実行するものに分けます。

> 2026-05-30 実装メモ: Codex 自動テストとして `npm test` を追加済みです。現在は backend smoke test と frontend build をまとめて実行します。

## 前提

- 対象状態: `codex/provisional-snapshot`
- backend: Express
- DB: 開発時は SQLite
- frontend: Vite + React
- GUI テストは `npm run dev` で起動したアプリを対象にする。
- 自動テストは DB を reset してから実行し、テスト間で状態を共有しない。

## テスト環境

### ローカル起動

```powershell
Copy-Item .env.example .env
npm install
npm run setup
npm run db:reset
npm run dev
```

## ユーザーが GUI で行うテスト

### U-01 一般ユーザー登録

対象: 一般ユーザー

手順

1. ログイン画面で新規登録を選ぶ。
2. 名前、メールアドレス、パスワード、年代、ユーザー区分を入力する。
3. 登録する。
4. 登録後にホームへ遷移することを確認する。
5. アカウント画面で登録情報が表示されることを確認する。

期待結果

- 登録後にログイン済み状態になる。
- ポイント残高が表示される。
- `npm run db:report` で `users` に新規ユーザーが増えている。

### U-02 一般ユーザーログイン、ログアウト

対象: 一般ユーザー

手順

1. `demo@example.com` / `password123` でログインする。
2. ホーム画面に遷移することを確認する。
3. ブラウザをリロードする。
4. ログイン状態が復元されることを確認する。
5. アカウント画面からログアウトする。

期待結果

- ログイン後にホームが表示される。
- リロード後もログイン状態が維持される。
- ログアウト後はログイン画面へ戻る。

### U-03 イベント参加

対象: 一般ユーザー

手順

1. ログインする。
2. イベント画面へ移動する。
3. 未参加イベントの参加操作を行う。
4. ホームまたはイベント画面でポイントが増えていることを確認する。
5. 参加履歴タブで参加したイベントが表示されることを確認する。
6. 同じイベントを再度参加しようとする。

期待結果

- 初回参加でポイントが増える。
- 参加履歴にイベントが追加される。
- 同一イベントの重複参加はできない。
- `npm run db:report` で `participations` と `point_transactions` に記録が増える。

### U-04 ポイント交換

対象: 一般ユーザー

手順

1. ログインする。
2. ウォレット画面へ移動する。
3. 交換候補からサービスを選んで交換する。
4. ポイント残高が減っていることを確認する。
5. 交換履歴に記録が表示されることを確認する。

期待結果

- 交換後にポイントが減る。
- 交換履歴に対象サービスが表示される。
- `npm run db:report` で `point_transactions` に `exchange` が増える。

### U-05 ポイント不足時の交換抑止

対象: 一般ユーザー

手順

1. ポイントを消費して不足状態にする。
2. 必要ポイントが残高を超えるサービスを交換しようとする。

期待結果

- 交換できない。
- ポイント残高が変わらない。
- エラー表示が確認できる。

## Codex が自動で行うテスト

### C-01 静的検証

実行タイミング: 毎回

内容

- `npm --prefix frontend run build`
- TypeScript 型チェック
- Vite build
- package lock の整合確認

合格条件

- build が成功する。
- TypeScript エラーがない。

### C-02 backend 起動 smoke

実行タイミング: backend 変更時

内容

- `npm run db:reset`
- backend を空き port で起動する。
- `GET /` を実行する。
- `npm run db:report` を実行する。

合格条件

- API が `Link Town Backend API is running.` を返す。
- seed ユーザー、イベント、店舗、サービスが作成される。

### C-03 認証 API テスト

実行タイミング: auth 変更時

内容

- 一般ユーザー登録成功
- メール重複登録失敗
- 一般ユーザーログイン成功
- 誤パスワードログイン失敗
- 管理者ログイン成功
- 誤管理者ログイン失敗

合格条件

- 正常系は token を返す。
- 異常系は 400 または 401 を返す。
- パスワード hash が平文保存されていない。

### C-04 一般ユーザー API テスト

実行タイミング: users 変更時

内容

- 自分のポイント取得成功
- 他人のポイント取得失敗
- 自分の履歴取得成功
- 他人の履歴取得失敗
- 管理者 token で任意ユーザー取得成功

合格条件

- 権限境界が維持される。
- 履歴 response shape が frontend 型と一致する。

### C-05 イベント参加 API テスト

実行タイミング: events 変更時

内容

- イベント一覧取得成功
- イベント参加成功
- ポイント残高増加
- 参加履歴追加
- point transaction 追加
- 同一イベント重複参加失敗
- 不正 event_id 失敗

合格条件

- 成功時のみ DB が更新される。
- 重複参加ではポイントが増えない。

### C-06 ポイント交換 API テスト

実行タイミング: points 変更時

内容

- サービス一覧取得成功
- ポイント交換成功
- ポイント残高減少
- exchange transaction 追加
- ポイント不足時の交換失敗
- 不正 service_id 失敗

合格条件

- 成功時のみ DB が更新される。
- ポイント不足では残高が変わらない。

### C-07 管理者 API テスト

実行タイミング: admin 変更時

内容

- 一般ユーザー token で admin API が失敗する。
- 管理者 token でイベント作成、更新が成功する。
- 管理者 token で店舗作成、更新が成功する。
- 管理者 token でサービス作成が成功する。
- 統計取得が成功する。

合格条件

- 管理者 API は admin role のみ成功する。
- 作成したデータが一般ユーザー API から参照できる。

### C-08 frontend component テスト

実行タイミング: frontend 変更時

内容

- ログイン画面の入力、送信
- 新規登録画面の入力、送信
- ホームの loading、error、empty、success 表示
- イベント一覧の参加済み状態
- ウォレットのポイント不足状態
- アカウント画面のユーザー情報表示

合格条件

- API mock に応じた表示が変化する。
- 主要ボタンが意図した handler を呼ぶ。

### C-09 Playwright e2e

実行タイミング: UI 導線変更時

内容

- DB reset
- backend 起動
- frontend 起動
- 一般ユーザーでログイン
- イベント参加
- ポイント交換
- 履歴確認
- ログアウト
- 管理者でログイン
- イベント作成
- 一般ユーザー側でイベント反映確認

合格条件

- GUI 操作で DB が期待通り変化する。
- 主要導線で console error が出ない。
- desktop と mobile viewport で表示崩れがない。

### C-10 QR e2e

実行タイミング: QR 実装後

内容

- 有効 QR payload を生成する。
- Playwright で QR 入力または mock camera 経由の読み取りを行う。
- 参加成功を確認する。
- 同じ QR を再実行して重複参加失敗を確認する。
- 無効 QR で失敗を確認する。

合格条件

- 有効 QR だけ DB が更新される。
- 重複と無効 QR ではポイントが増えない。

### C-11 セキュリティ回帰テスト

実行タイミング: auth、env、deploy 変更時

内容

- JWT secret 未設定時の扱い確認
- token なし API の 401 確認
- role 不一致の 403 確認
- password policy の確認
- rate limit の確認
- `.env` が commit 対象に入らないことの確認

合格条件

- 認証なし、権限なしのアクセスが拒否される。
- 開発用 seed や secret が本番前提の設定に混入しない。

### C-12 migration / DB 再現テスト

実行タイミング: DB schema 変更時

内容

- 空 DB から schema 作成
- seed 作成
- 既存 DB への migration
- rollback 可能性の確認
- MySQL と SQLite の差異確認

合格条件

- 新規環境で DB が再現できる。
- 既存データを壊さず schema 更新できる。
