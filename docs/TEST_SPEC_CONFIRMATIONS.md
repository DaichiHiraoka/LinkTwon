# テスト仕様書 実施方針

対象は `outputs/test-spec-20260614/D1_テスト仕様書_20260614.xlsx` の T-001 から T-060 です。
アプリの API、画面、管理 CRUD、QR ポータル、翻訳キャッシュの実装と自動スモークテストは用意済みです。以下は、テスト実施時の決定事項と残る受入確認です。

## 1. ポイント購入の決済方式

- 対象: T-025, T-026
- 現状: ポイント購入の記録、成功時の残高加算、失敗時に残高を変えない API はある。自動テストでは `simulate_status` で成功と失敗を再現できる。
- 決定: モック決済を継続する。
- 実施方法: 通常のモック支払い方法は成功し、ブランドが `mock-fail` の支払い方法は失敗として記録する。失敗時は購入履歴だけを作成し、ポイント残高を変更しない。

## 2. メール送信先と再設定テストの方式

- 対象: T-005, T-006
- 現状: SMTP、Resend、開発用 outbox を実装済み。テスト環境では outbox とトークン公開で検証できる。
- 決定: デプロイ済みのメール送信基盤を使い、実メールで確認する。
- 実施前確認: Render の `MAIL_DRIVER`、送信プロバイダの認証情報、`SMTP_FROM` を確認し、テスト送信先を指定して再設定メールの到達を確認する。

## 3. 翻訳キャッシュの定期更新の実行基盤

- 対象: T-051, T-052, T-053
- 決定: Render Cron Job を30分ごと（`*/30 * * * *`、UTC）に実行する。
- 実施方法: Cron Job は `POST /api/translations/refresh` を呼ぶ。Backend と Cron Job の両方に同じ `TRANSLATION_REFRESH_KEY` を設定する。
- 注意: Render Cron Job は最低月額 1 ドルで、既存 Blueprint の `sync: false` な秘密値は Dashboard で手動設定が必要である。

## 4. デプロイ済み環境で使うテスト用データベース

- 対象: 状態を変更する T-001, T-005-T-006, T-015-T-020, T-025-T-031, T-037-T-041, T-044-T-046, T-049-T-060
- 現状: ローカル自動テストは一時 SQLite データベースで完結する。デプロイ済み Backend は Aiven MySQL を使う構成で、テスト実行によるユーザー、参加履歴、ポイント、通知、問い合わせ、翻訳キャッシュの変更を分離していない。
- 決定: デプロイ済みの Aiven DB を使う。プロトタイプ用途のため、テスト副作用を許容する。
- 実施方法: テスト用ユーザー・イベント・店舗には `test_` プレフィックスを付け、テスト後に必要なら管理画面から削除する。

## 5. 実機カメラを使う受入テスト端末

- 対象: T-014, T-044, T-047, T-049
- 現状: モーダルの外側タップ・下フリック、カメラ読取、手入力フォールバックは実装済み。自動スモークテストでは手入力 QR を検証している。
- 方針: この項目は受入テスト時に実機で確認する。実装上の追加対応は行わない。
